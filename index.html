Section 0 — Core Concepts

Algorithms, String Matching & Why We Need Hashing

This section requires no prior knowledge. If you are reading about string matching algorithms for the first time, start here. Every term is defined before it is used.

What is an Algorithm?

An algorithm is a finite, step-by-step procedure that takes an input, performs a series of well-defined operations, and produces an output. Two algorithms can solve the same problem but have very different performance — choosing the right one matters enormously in practice.

For string matching, both the naive brute-force algorithm and Rabin-Karp find all occurrences of a pattern in a text. But for a 1,000,000-character document and a 100-character pattern, brute-force may do 100,000,000 comparisons while Rabin-Karp does approximately 1,000,100. That is a 1000× speedup.

What is the String Matching Problem?

Given a text T of length n and a shorter pattern P of length m, find every starting position (called a shift) where P occurs in T. This is the algorithm behind every Ctrl+F search, every DNA sequence scanner, every spam filter, and every plagiarism detector.

Text T    = "AABAACAADAABAABA"    (n = 16)
Pattern P = "AABA"               (m = 4)
Matches   →  shift s = 0, s = 9, s = 12

The output is the list of all valid shifts. The challenge is computing this list efficiently.

Why is Naive Brute-Force Slow?

The simplest method slides P across T one position at a time and compares characters one by one. At each of the n positions, it may compare up to m characters — giving a worst-case cost of O(n × m).

for s = 0 to n-m:           // n positions
    if T[s..s+m-1] == P:    // m comparisons each
        report match at s   // total: O(n × m)

For n=1,000,000 and m=100 that is 100 million character comparisons. Rabin-Karp reduces this to approximately O(n + m) by replacing character-by-character comparisons with a single integer comparison using a hash.

What is a Hash Function?

A hash function converts a string (or any data) into a single number called a hash value. A good hash function has two key properties:

  • Consistency: The same string always produces the same hash value.
  • Discrimination: Different strings very rarely produce the same hash value.

Because of the consistency property, if two strings have different hash values, they are guaranteed to be different strings. This lets us reject windows instantly with a single integer comparison — no character reading needed.

What is Big-O Notation?

Big-O notation describes how the running time of an algorithm grows as the input size increases. It ignores constants and focuses on the dominant term, because for large inputs the growth rate is what matters.

NotationNameMeaningExample
O(1)ConstantSame time regardless of input sizeInteger comparison
O(m)Linear in mGrows proportionally with pattern lengthComputing one hash
O(n)Linear in nGrows proportionally with text lengthOne pass over text
O(n + m)LinearGrows with both; dominated by n for large textsRabin-Karp expected
O(nm)QuadraticGrows with product of both — much worseBrute-force worst case

Key rule: O(1) < O(m) < O(n) < O(n+m) < O(nm). The further right, the slower the algorithm grows for large inputs. Always prefer algorithms further to the left.

Section 2 — Algorithm Overview

What is Rabin-Karp? How Does It Work?

What problem does it solve?

Imagine you are searching for the word "AABA" inside the sentence "AABAACAADAABAABA". You want to find every position where it appears. This is the string matching problem: given a text T of length n and a pattern P of length m, find all starting positions where P occurs in T.

This sounds simple, but doing it efficiently is a core challenge in computer science. Every text editor's Ctrl+F, every DNA sequence search, every plagiarism detector solves exactly this problem billions of times.

Why not just compare directly?

The simplest approach — called Naive / Brute-Force — slides the pattern across the text one position at a time and compares every character. For each of the n positions, it may compare up to m characters. This gives a worst-case cost of O(n × m). For large texts and patterns, this is unacceptably slow.

Rabin-Karp's key idea: instead of comparing all m characters at each position, first compare a single number. That number — called a hash — summarises the entire window in one integer. Two identical strings always have the same hash, so if the hashes differ, we can skip the window entirely without looking at a single character.

What is a hash function?

A hash function turns a string into a number. A good hash function spreads different strings across different numbers, so two strings that are NOT equal are very unlikely to produce the same hash. Rabin-Karp uses a polynomial hash:

h("ABC") = (A × d² + B × d¹ + C × d⁰) mod q

where:
  d = base (usually alphabet size, e.g. 26 or 256)
  q = a prime number (e.g. 101) to keep numbers small
  A, B, C = numeric values of each character (e.g. ASCII codes)

The modulo operation (mod q) keeps the number small so it fits in a machine integer. It also means two different strings CAN produce the same hash — this is called a collision and is unavoidable with finite-size integers. We will deal with this below.

The rolling hash — why it is clever

The expensive part is computing the hash of each window. If we recomputed it from scratch each time (looking at all m characters), we gain nothing. The brilliant part of Rabin-Karp is the rolling hash: when the window slides one position right, we can update the hash in O(1) — constant time, regardless of m.

The intuition: the new window shares m−1 characters with the old window. We just remove the contribution of the character that left, and add the contribution of the new character that entered:

// Old window: T[s .. s+m-1]    New window: T[s+1 .. s+m]
// They share: T[s+1], T[s+2], ..., T[s+m-1]  (m-1 characters)

h_new = (d × (h_old − T[s] × h_high) + T[s+m]) mod q

// h_high = d^(m-1) mod q  (precomputed once before the loop)
// T[s]   = character being removed  (the leftmost one leaving)
// T[s+m] = character being added    (the new rightmost one)
What are spurious hits?

A spurious hit occurs when h_window = h_pattern but the window and pattern are actually different strings — a hash collision. When this happens, we must verify character by character (O(m) cost). The algorithm is still correct — we never miss a match and never report a false match — but we pay extra time for the verification.

With a prime q = 101, the probability of a spurious hit at any given window is roughly 1/101 ≈ 1%. With q = 1000003 it drops to 0.0001%. In the extreme (adversarially crafted) input, every window is a spurious hit — giving O(nm) worst case. In practice, spurious hits are very rare.

When is Rabin-Karp the right choice?

Best for: Searching for many patterns simultaneously (store all hashes in a set; one O(n) pass finds any of them). Plagiarism detection, DNA motif search, network intrusion detection.
Not ideal for: Single-pattern search where worst-case O(nm) is unacceptable — use KMP (guaranteed O(n)) instead.
Unique advantage: O(1) space (no auxiliary table like KMP's failure function).

Most common mistake

Forgetting to apply mod q to every arithmetic operation. Without it, intermediate values grow exponentially and overflow, producing garbage hash values. Every add, multiply, and subtract must be followed by mod q — especially the subtraction in the rolling formula, which can go negative (add q before taking mod to fix this).

Key vocabulary to know

Text (T): The large string we are searching through.
Pattern (P): The shorter string we are looking for.
Window: The m-character substring of T currently being examined.
Shift (s): The starting position of the current window.
Hash: A number that represents a string, computed by a hash function.
Rolling hash: A technique to update a window's hash in O(1) as it slides.
Spurious hit: A false positive where hashes match but strings do not.
h_high: The precomputed value d^(m-1) mod q, used in the rolling formula.

Check your understanding
  1. Why does Rabin-Karp compute h(P) only once but compute h(T[0..m-1]) once as well? Could we compute h(T[0..m-1]) inside the loop?
  2. If two windows have the same hash but different characters, what does Rabin-Karp do? What would happen if it just reported a match anyway?
  3. If you increase q from 101 to 1009, what effect does this have on the number of spurious hits? What is the tradeoff?

Most common implementation mistake

The rolling hash subtraction can produce a negative number. In most programming languages, the % operator on a negative number returns a negative result. Always add q before taking mod: ((h % q) + q) % q. Forgetting this will cause your algorithm to miss matches silently.

Section 3 — Worked Example

Tracing Rabin-Karp by Hand — Every Calculation Shown

We will trace through the full algorithm by hand, step by step, computing every hash value manually. This is the best way to see exactly how the rolling hash works in practice.

Setup: Text T = "ABCABD" (n=6), Pattern P = "ABD" (m=3), d = 10, q = 13.
Character values: A=1, B=2, C=3, D=4. h_high = d^(m-1) mod q = 10² mod 13 = 100 mod 13 = 9.

1

Compute h(P)

h("ABD") = (1·10² + 2·10 + 4) mod 13 = (100+20+4) mod 13 = 124 mod 13 = 7. Also h_high = d^(m-1) mod q = 10² mod 13 = 100 mod 13 = 9.

2

First window "ABC" → hash = ?

h("ABC") = (1·100 + 2·10 + 3) mod 13 = 123 mod 13 = 6. Compare: 6 ≠ 7. Hash mismatch — skip, no character check needed.

3

Roll to "BCA"

h_new = (10·(6 − 1·9) + 3) mod 13 = (10·(−3) + 3) mod 13 = (−27) mod 13 = 12. (We add 13 to keep positive.) Compare: 12 ≠ 7. Skip.

4

Roll to "CAB"

h_new = (10·(12 − 2·9) + 2) mod 13 = (10·(−6) + 2) mod 13 = (−58) mod 13 = 7 (add 65). Compare: 7 == 7. Hash match! Verify: C≠A → spurious hit. No match.

5

Roll to "ABD"

h_new = (10·(7 − 3·9) + 4) mod 13 = (10·(−20) + 4) mod 13 = (−196) mod 13 = 7. Compare: 7 == 7. Verify: A==A, B==B, D==D → True match at s=3!

6

Takeaway

Step 4 showed a spurious hit — hashes matched but strings differed. Step 5 confirmed a true match after character verification. This is the full Rabin-Karp cycle in action.

Why trace by hand? Working through an example manually — computing each hash value yourself — is the single most effective way to understand how the rolling formula works. Read each step, then verify the arithmetic on paper before moving on. This is exactly what the tool shows you when you press Step.

Now try a new example yourself
  1. Set T = "ABCABD", P = "ABD", d = 10, q = 13 in the tool. Press Build Steps and Step through. Does the tool show the same hash values you computed above?
  2. Now change q = 7 (a smaller prime). Does the spurious hit at step 4 still appear? Does a new one appear?
  3. What is h_high for m=4 and d=10 and q=13? (Answer: 10³ mod 13 = 1000 mod 13 = 12.)

Common arithmetic mistake in the rolling formula

When computing (h_old − T[s] × h_high), the subtraction can be negative. Example: h_old=6, T[s]×h_high=9 gives −3. You must add q before taking mod: (−3 + 13) mod 13 = 10. Never skip this step.

Full Complexity Derivation — step by step

▼

A derivation does not just state "O(n+m)" — it shows exactly why by counting the dominant operations. Here is the full argument:

1

Preprocessing cost — O(m)

We perform three one-time operations before the main loop:
• Compute h_high = d^(m-1) mod q: one loop of m-1 multiplications → O(m)
• Compute h(P): one loop of m multiplications → O(m)
• Compute h(T[0..m-1]): one loop of m multiplications → O(m)
Total preprocessing: 3 × O(m) = O(m).

2

Main loop iterations — O(n)

The loop runs for s = 0, 1, 2, ..., n-m. That is exactly n − m + 1 iterations. Since m ≤ n, this is at most n iterations → O(n).

3

Cost per loop iteration — O(1) in the expected case

At each shift s, we do:
• One integer comparison h_win == h_pat → O(1)
• One rolling hash update → O(1) (three arithmetic operations)
If the hash comparison fails (the common case), we move on. No character checking.
→ Per-iteration cost in the expected case: O(1).

4

Character verification — only on hash matches

When h_win == h_pat, we run a character-by-character check costing O(m). This happens:
• At every true match (unavoidable — we must verify). If there are k matches, this costs k × O(m).
• At every spurious hit. With prime q, the probability of a spurious hit at any position is ≤ 1/q. Expected number of spurious hits ≤ n/q, so expected verification cost ≤ (n/q) × O(m).
For large q, this is negligible. Total expected verification: O(km + nm/q).

5

Putting it together — expected total

Total = Preprocessing + Loop + Verification
= O(m) + O(n) × O(1) + O(km + nm/q)
= O(m) + O(n) + O(nm/q)
For constant k and large q (e.g. q > n), the verification term → O(m).
→ Expected total: O(n + m)

6

Worst case — O(nm)

In the worst case, every window is a spurious hit. This happens when the hash function maps all possible m-character strings to the same value — for example, searching for "AAAB" in "AAAAAAAAA" with a poorly chosen (d, q) pair. Each of the n-m+1 shifts triggers O(m) verification. Total: O(nm). Load the "All-A spurious" preset in the tool to observe this.

Space complexity — O(1)

The algorithm uses only a fixed number of integer variables (h_pat, h_win, h_high, s, j, n, m, d, q) regardless of the input size. Unlike KMP, which requires an O(m) failure table, Rabin-Karp needs no auxiliary data structure. Space complexity: O(1).

Section 5 — CLO-2 · Searching Role

Hashing as a Search Filter — Two-Tier Strategy

The central searching mechanism in Rabin-Karp is the hash comparison that acts as a fast filter before expensive character verification. At every window shift, O(1) hash arithmetic decides whether to skip the window entirely or to run a full O(m) character scan.

This is a two-tier search strategy:

Tier 1 — Hash filter (O(1) per window)

Compare h_window with h_pattern. If they differ, skip immediately. This handles the vast majority of positions in a typical search.

Tier 2 — Character verification (O(m), only on hash match)

When hashes match, verify character by character to rule out spurious hits. In the expected case this happens only at true match positions.

What happens without hashing?

Without the hash filter, every window requires a full O(m) character scan — exactly the naive brute-force approach with O(nm) worst case. The hash is what makes the expected case O(n + m).

Advantage

Skip most windows instantly

With a prime q = 101, roughly 99% of non-match windows are rejected after a single integer comparison.

Risk

False positives (spurious hits)

Hash collisions cause spurious work. Choosing a larger prime (e.g., q = 1000003) reduces collision rate to under 0.0001%.

Edge Cases, Applicability & When NOT to Use It

Edge cases to know
E1

Pattern longer than text (m > n)

Impossible to match — the algorithm should return immediately with no matches. The loop condition s ≤ n-m would have no iterations. Try typing a longer pattern than text in the tool and pressing Build Steps.

E2

Pattern equals text (m = n)

Only one window exists (s = 0 only). One hash comparison, one possible verification. The algorithm still works correctly — it is just a single-step search.

E3

Overlapping matches

Rabin-Karp finds ALL occurrences, including overlapping ones. For example, searching for "AA" in "AAAA" finds matches at s=0, 1, and 2. The tool's match chips will show all three positions.

E4

No match anywhere

Every window produces a hash mismatch (or spurious hit followed by failed verification). The match list stays empty. Load the "No match" preset to see this — every window is highlighted red.

E5

Impact of base (d) and prime (q) choice

A small prime (e.g., q = 7) causes many collisions and many spurious hits — the algorithm slows toward O(nm). A large prime (e.g., q = 1000003) minimises collisions but requires careful handling of large intermediate values. Try changing the q field in the tool and compare the spurious hit counter.

When Rabin-Karp is the right choice

Use Rabin-Karp when...

• You are searching for multiple patterns at once (e.g., 1000 spam phrases in an email — one pass, hash set lookup).
• The text is large but matches are rare — the hash filter eliminates almost all windows cheaply.
• Memory is constrained — O(1) space vs KMP's O(m) failure table.
• You need a simple implementation — Rabin-Karp is shorter and easier to code correctly than KMP.

Do NOT use Rabin-Karp when...

• You need a worst-case guarantee — adversarial inputs can degrade it to O(nm). Use KMP or Boyer-Moore instead.
• The alphabet is very small (e.g., binary strings) — collision rate is high with small alphabets unless q is very large.
• The pattern is extremely short (m = 1 or 2) — the rolling hash overhead is not worth it; a simple linear scan suffices.

Section 6 — CLO-3 · Design Paradigm

Rolling Hash — Incremental Computation & Design Correctness

Rabin-Karp's design paradigm is incremental computation (a variant of the divide-and-conquer / sliding-window family). The key insight is: instead of recomputing the window hash from scratch at every shift (which costs O(m)), we exploit the structural relationship between adjacent windows.

The rolling hash relationship

Window at s: T[s], T[s+1], ..., T[s+m-1]
Window at s+1: T[s+1], T[s+2], ..., T[s+m]
The two windows share m-1 characters. The rolling formula reuses the previous hash, subtracts the contribution of the leaving character, and adds the entering character — all in O(1).

// Naïve recomputation — O(m) per window
h_win = 0
for j = 0 to m-1:
    h_win = (h_win * d + T[s+j]) % q     // m multiplications

// Rolling update — O(1) per window
// Remove leaving char, shift left, add entering char
h_win = (d * (h_win - T[s] * h_high) + T[s+m]) % q

The rolling formula is correct because polynomial evaluation is linear: d · (h − T[s] · d^(m-1)) shifts the polynomial left by one degree (equivalent to removing the highest-degree term and multiplying the rest by d), then adding T[s+m] appends the new term at degree 0.

Comparison with a different approach: A Bloom-filter approach would use multiple hash functions to reduce false positives, but each update still requires recomputing multiple hashes. The rolling hash achieves a single O(1) update per window while maintaining the filtering benefit — making it strictly more efficient for single-pattern search.

Counterexample: Why naive recomputation fails at scale

For n = 1,000,000 and m = 100, naive recomputation costs 100,000,000 multiplications per search. Rolling hash costs exactly 1 multiply + 2 add/subtracts per window. At scale, the difference is the boundary between a usable tool and an unusable one.

Paradigm

Sliding Window

Maintains a fixed-size window over the input, updating state incrementally rather than recomputing from scratch. O(1) amortized update is the hallmark of this paradigm.

Key insight

Structural reuse

Adjacent windows share m-1 characters. Exploiting this overlap reduces per-step cost from O(m) to O(1) — the same principle behind Kadane's algorithm and sliding-window maximum.

Correctness Proof — Why Rabin-Karp Never Misses a Match

Why is the algorithm always correct?

Correctness means: the algorithm reports a match at position s if and only if T[s..s+m-1] = P. We need to prove two things:

→

No true match is ever missed

Suppose T[s..s+m-1] = P (a true match exists). Since the hash function is deterministic, identical strings always produce identical hashes. Therefore h_win = h_pat. The condition h_win == h_pat is satisfied, and we enter the character verification loop. Since the strings are identical, every character matches, and we correctly report a match. No true match is ever skipped.

→

No false match is ever reported

Suppose h_win = h_pat but T[s..s+m-1] ≠ P (a spurious hit). The character verification loop will find a mismatch at some position j and terminate early without reporting a match. A match is only reported after every character is confirmed equal.

→

The rolling hash preserves correctness

We must verify the rolling formula is mathematically equivalent to recomputing from scratch. Let h(s) = Σ T[s+j] × d^(m-1-j) mod q for j = 0..m-1. Then:
h(s+1) = Σ T[s+1+j] × d^(m-1-j) mod q
= (h(s) − T[s] × d^(m-1)) × d + T[s+m] mod q
This is exactly the rolling formula. Every update is algebraically exact, so h_win always equals the correct hash of the current window.

What breaks if you change the design?

What if you skip character verification on hash matches?

The algorithm becomes incorrect. Spurious hits would be reported as matches. You would get false positives — positions reported as matches where T[s..s+m-1] ≠ P. Character verification is not optional; it is what makes the algorithm correct despite collisions.

What if you do not use mod q?

Intermediate hash values grow as d^m, which for d=256 and m=20 exceeds 10^48 — far beyond any integer type. Overflow causes wrap-around, producing wrong hash values and missing true matches. The mod operation is essential for correctness in finite-precision arithmetic.

What if you recompute h_win from scratch each time instead of rolling?

The algorithm remains correct but loses its efficiency. Each window would cost O(m) to hash, making the total O(nm) — the same as naive brute force. The rolling hash is the design decision that makes Rabin-Karp O(n+m) in the expected case. It is a performance optimisation, not a correctness requirement.

Section 7 — Common Mistakes & Self-Study Challenges

What Goes Wrong — And How to Avoid It

The 5 Most Common Implementation Mistakes

Mistake 1 — Forgetting mod q on every operation

Every addition, multiplication, AND subtraction in the hash formula must be followed by % q. Without it, intermediate values grow exponentially (dm for d=256, m=30 exceeds 1072) and overflow any integer type silently.

Mistake 2 — Negative result from the rolling subtraction

The expression (h - T[s] * h_high) % q can be negative in languages where % returns the sign of the dividend. Always use ((h - T[s]*h_high) % q + q) % q.

Mistake 3 — Reporting a match on hash equality without verifying

Never report a match just because h_win == h_pat. Hash collisions (spurious hits) are rare but real. Always perform the O(m) character-by-character verification before reporting. Skipping this makes the algorithm incorrect.

Mistake 4 — Computing h_high inside the loop

h_high = dm-1 mod q must be computed ONCE before the loop begins, not at every iteration. Computing it inside the loop adds O(m) cost per window, destroying the O(1) rolling update benefit.

Mistake 5 — Using a non-prime or too-small q

A small q (e.g. q=7) causes many hash collisions, forcing character verification far too often and degrading performance toward O(nm). Always use a large prime (101, 1009, 1000003). Using a composite number as q does not evenly distribute hash values and increases the collision rate.
