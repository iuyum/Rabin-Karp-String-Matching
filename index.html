Algorithm Overview

▼

Rabin-Karp is a string-searching algorithm invented by Michael Rabin and Richard Karp in 1987. Given a text T of length n and a pattern P of length m, it finds every position in T where P occurs. The key innovation is using a hash function to speed up comparisons.

Instead of comparing characters at each window position, the algorithm computes a polynomial rolling hash of the pattern once, then slides a same-length hash window across the text. At each step, comparing two integers (O(1)) replaces comparing m characters (O(m)). The rolling hash updates in O(1) as the window moves — so the total expected cost is O(n + m).

The unavoidable weakness is hash collisions: two different strings can share the same hash. When this happens (a spurious hit), a full O(m) character verification is required. With a good prime modulus, spurious hits are rare. In adversarial inputs, however, they degrade performance to O(nm).

When to use Rabin-Karp

Rabin-Karp is the best choice when searching for multiple patterns simultaneously — store all pattern hashes in a hash set, then one O(n) pass detects any of them. For single-pattern search where worst-case guarantees matter, KMP is preferable.

Common mistake

Forgetting to take all hash arithmetic modulo q causes integer overflow and produces incorrect results. Every addition, multiplication, and subtraction must be reduced mod q.

Worked Example — step by step

▼

Text T = "ABCABD", Pattern P = "ABD", d = 10, q = 13. Assign A=1, B=2, C=3, D=4.

1

Compute h(P)

h("ABD") = (1·10² + 2·10 + 4) mod 13 = (100+20+4) mod 13 = 124 mod 13 = 7. Also h_high = d^(m-1) mod q = 10² mod 13 = 100 mod 13 = 9.

2

First window "ABC" → hash = ?

h("ABC") = (1·100 + 2·10 + 3) mod 13 = 123 mod 13 = 6. Compare: 6 ≠ 7. Hash mismatch — skip, no character check needed.

3

Roll to "BCA"

h_new = (10·(6 − 1·9) + 3) mod 13 = (10·(−3) + 3) mod 13 = (−27) mod 13 = 12. (We add 13 to keep positive.) Compare: 12 ≠ 7. Skip.

4

Roll to "CAB"

h_new = (10·(12 − 2·9) + 2) mod 13 = (10·(−6) + 2) mod 13 = (−58) mod 13 = 7 (add 65). Compare: 7 == 7. Hash match! Verify: C≠A → spurious hit. No match.

5

Roll to "ABD"

h_new = (10·(7 − 3·9) + 4) mod 13 = (10·(−20) + 4) mod 13 = (−196) mod 13 = 7. Compare: 7 == 7. Verify: A==A, B==B, D==D → True match at s=3!

6

Takeaway

Step 4 showed a spurious hit — hashes matched but strings differed. Step 5 confirmed a true match after character verification. This is the full Rabin-Karp cycle in action.

CLO-2 — Searching Role

Hashing as a Search Filter

The central searching mechanism in Rabin-Karp is the hash comparison that acts as a fast filter before expensive character verification. At every window shift, O(1) hash arithmetic decides whether to skip the window entirely or to run a full O(m) character scan.

This is a two-tier search strategy:

Tier 1 — Hash filter (O(1) per window)

Compare h_window with h_pattern. If they differ, skip immediately. This handles the vast majority of positions in a typical search.

Tier 2 — Character verification (O(m), only on hash match)

When hashes match, verify character by character to rule out spurious hits. In the expected case this happens only at true match positions.

What happens without hashing?

Without the hash filter, every window requires a full O(m) character scan — exactly the naive brute-force approach with O(nm) worst case. The hash is what makes the expected case O(n + m).

Advantage

Skip most windows instantly

With a prime q = 101, roughly 99% of non-match windows are rejected after a single integer comparison.

Risk

False positives (spurious hits)

Hash collisions cause spurious work. Choosing a larger prime (e.g., q = 1000003) reduces collision rate to under 0.0001%.

CLO-3 — Design Paradigm

Rolling Hash — Incremental Computation Strategy

Rabin-Karp's design paradigm is incremental computation (a variant of the divide-and-conquer / sliding-window family). The key insight is: instead of recomputing the window hash from scratch at every shift (which costs O(m)), we exploit the structural relationship between adjacent windows.

The rolling hash relationship

Window at s: T[s], T[s+1], ..., T[s+m-1]
Window at s+1: T[s+1], T[s+2], ..., T[s+m]
The two windows share m-1 characters. The rolling formula reuses the previous hash, subtracts the contribution of the leaving character, and adds the entering character — all in O(1).

// Naïve recomputation — O(m) per window
h_win = 0
for j = 0 to m-1:
    h_win = (h_win * d + T[s+j]) % q     // m multiplications

// Rolling update — O(1) per window
// Remove leaving char, shift left, add entering char
h_win = (d * (h_win - T[s] * h_high) + T[s+m]) % q

The rolling formula is correct because polynomial evaluation is linear: d · (h − T[s] · d^(m-1)) shifts the polynomial left by one degree (equivalent to removing the highest-degree term and multiplying the rest by d), then adding T[s+m] appends the new term at degree 0.

Comparison with a different approach: A Bloom-filter approach would use multiple hash functions to reduce false positives, but each update still requires recomputing multiple hashes. The rolling hash achieves a single O(1) update per window while maintaining the filtering benefit — making it strictly more efficient for single-pattern search.

Counterexample: Why naive recomputation fails at scale

For n = 1,000,000 and m = 100, naive recomputation costs 100,000,000 multiplications per search. Rolling hash costs exactly 1 multiply + 2 add/subtracts per window. At scale, the difference is the boundary between a usable tool and an unusable one.

Paradigm

Sliding Window

Maintains a fixed-size window over the input, updating state incrementally rather than recomputing from scratch. O(1) amortized update is the hallmark of this paradigm.

Key insight

Structural reuse

Adjacent windows share m-1 characters. Exploiting this overlap is what reduces per-step cost from O(m) to O(1) — the same principle behind Kadane's algorithm and sliding-window maximum.
